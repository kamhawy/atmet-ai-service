Azure.AI.Projects 2.0 + Azure.AI.Extensions.OpenAI 2.0 use ProjectOpenAIClient
(not projectClient.OpenAI). See the runnable harness:

  src/tools/ATMET.AI.WorkflowSpike/README.md

Minimal pattern (C#):

  using Azure.AI.Extensions.OpenAI;
  using Azure.AI.Projects;
  using Azure.Identity;

  var projectClient = new AIProjectClient(new Uri(endpoint), new DefaultAzureCredential());
  var openAi = projectClient.ProjectOpenAIClient;
  var conversation = (await openAi.GetProjectConversationsClient()
      .CreateProjectConversationAsync()).Value;
  var agentReference = new AgentReference(agentName, agentVersion);
  var responseClient = openAi.GetProjectResponsesClientForAgent(agentReference, conversation);
  var r1 = (await responseClient.CreateResponseAsync(userMessage)).Value;
  // Resume / next turn: chain with previous_response_id (SDK overload or CreateResponseOptions).
  var r2 = (await responseClient.CreateResponseAsync("Follow-up", previousResponseId: r1.Id)).Value;
  var text = r2.GetOutputText();

Packages (match ATMET.AI.Api):
  Azure.AI.Projects 2.0.0
  Azure.AI.Extensions.OpenAI 2.0.0
  Azure.Identity

Align with the API host configuration (appsettings.json section "AzureAI"):
  ProjectEndpoint, WorkflowAgentName, WorkflowAgentVersion
  On Azure App Service / containers use the same flattened env vars as ASP.NET Core:
  AzureAI__ProjectEndpoint, AzureAI__WorkflowAgentName, AzureAI__WorkflowAgentVersion

The WorkflowSpike tool loads those files/variables automatically; see src/tools/ATMET.AI.WorkflowSpike/README.md.
